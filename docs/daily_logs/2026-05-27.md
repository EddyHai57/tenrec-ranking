## 设计审阅结论

采纳 Opus 对本地训练层的补充意见：

- 不依赖 `shuffle_buffer_size` 做局部混洗；物化 train 文件按 user 排列，训练前生成 deterministic hash-bucket shuffled train CSV。
- LR 必须是真正的线性模型：每个 categorical field 使用 `[vocab_size, 1]` 标量 lookup，求和后加全局 bias。
- CTR baseline 不使用 `pos_weight`、class reweighting 或重采样，避免破坏 LogLoss 概率校准。
- `metadata_path` 必须支持 CLI 覆盖；服务器 full preprocessing 会生成新的 run_id，不能写死本地 smoke run。
- `max_train_rows` / `max_valid_rows` 是 head 截断，只允许 smoke，不允许报告正式指标。

## Tenrec 论文与官方 benchmark 审阅

审阅对象：

```text
Tenrec: A Large-scale Multipurpose Benchmark Dataset for Recommender Systems
Official GitHub: https://github.com/yuangh-x/2022-NIPS-Tenrec
Official CTR code: https://github.com/yuangh-x/2022-NIPS-Tenrec/blob/master/utils.py
```

关键事实：

- Tenrec 是 Tencent / Westlake University 等机构的 NeurIPS 2022 Datasets and Benchmarks 数据集。
- 论文确认数据来自四个场景：`QK-video`、`QK-article`、`QB-video`、`QB-article`。
- QK-video CTR task 使用 `QK-video-1M`，保留全部正样本，并按 positive:negative = 1:2 采样真实负样本。
- 论文 CTR 特征包含 `user_id`、`item_id`、`gender`、`age`、`video_category` 和 past 10 clicked items。
- 官方 `utils.py` 的 `ctrdataset()` 对全量 df 做 `LabelEncoder().fit_transform()`，并使用 `train_test_split(df, test_size=0.1)`。

判断：

- 不需要大改项目方向。Tenrec 仍适合作为本项目排序（Ranking）主数据。
- 数据本身不是主要问题；真正需要警惕的是官方 benchmark 协议和本项目可审计协议不同。
- 本项目 strict protocol 保持为主线：不负采样、user-order split、train-only vocab、AUC/GAUC/LogLoss。
- 后续可以增加 official-compatible reproduction protocol 作为对照，但不能替代 strict protocol。

## 多任务学习 Phase C 方向记录

根据 Tenrec 论文、官方 benchmark 和本地 `ctr_data_1M.csv` 行为分布，已将多任务学习（MTL / Multi-task Learning）记录为 Phase C 后期拔高方向。

已确认边界：

- MTL 架构是共享底座 + 多任务 head，不是多个任务完整模型并联，也不是 Two-Tower 召回结构。
- Tenrec 当前没有 purchase / conversion label，因此任务限定为 `click`、`like`、`share`、`follow`。
- 全量 `ctr_data_1M.csv` 正例率：
  - `click`: 28,880,860 / 120,342,306，约 24.0%。
  - `like`: 2,275,417 / 120,342,306，约 1.9%。
  - `share`: 250,089 / 120,342,306，约 0.2%。
  - `follow`: 179,788 / 120,342,306，约 0.15%。
- `share` 和 `follow` 极稀疏，per-task 指标预计高方差，后续必须同时报告有效样本数、正例数和 GAUC coverage。

Phase C 前置条件：

- 单任务 CTR 主线稳定。
- Phase B 的特征交互和 `hist_*` / user interest 使用口径完成。
- 多任务 label、loss 权重和 per-task metric 口径完成审计。

推进顺序：

```text
C1 Shared-Bottom -> C2 MMOE -> C3 PLE -> C4 ESMM
```

模型定位：

- Shared-Bottom：最小 MTL baseline，用于验证多任务共享表示；风险是 task conflict / negative transfer。
- MMOE：通过 task-specific gates 缓解任务冲突。
- PLE：通过 shared experts + task-specific experts 缓解 seesaw phenomenon。
- ESMM：仅作为 click -> like/share/follow 条件互动建模候选，用于探索稀疏反馈和样本选择偏差；Tenrec 无购买行为，不能写成 CVR / purchase 已完成方向。

- 尚未实现，不能写成项目成果或简历已完成内容。
